# Mirror-GUI Architecture

An overview of the mirror-gui architecture for contributors and developers.

## What Mirror-GUI Does

Mirror-GUI provides a web interface for building oc-mirror v2 `ImageSetConfiguration` files and executing mirror-to-disk operations. It lets users browse operator catalogs, select platform channels, compose configurations visually, and monitor mirroring progress — all without using the oc-mirror CLI directly.

The application is deployed as a container image that bundles the oc-mirror v2 binary, pre-fetched operator catalog metadata, and the web application.

## High-Level Flow

```
Browser (React SPA)
       |
   REST API / SSE
       |
Express Server (server/index.ts)
       |
       +--- Config Management         <- CRUD for ImageSetConfiguration YAML files
       |
       +--- Catalog Data              <- Pre-fetched operator metadata (offline browsing)
       |
       +--- Operation Execution       <- Spawns oc-mirror as child process
       |         |
       |     oc-mirror v2 CLI         <- Mirror-to-disk workflow
       |         |
       |     Log streaming (SSE)      <- Real-time log delivery to browser
       |
       +--- System Management         <- Pull secret, registry auth, cache, catalog sync
```

## Application Layers

### Frontend (React / PatternFly 6)

The frontend is a single-page application built with React 18 and PatternFly 6. Vite handles bundling for production and provides hot module replacement in development.

| Component | Purpose |
|-----------|---------|
| `Dashboard` | System overview: oc-mirror version, health status, pull secret detection, operation stats |
| `MirrorConfig` | Visual builder for `ImageSetConfiguration` YAML — platform channels, operators, additional images, YAML preview/edit, file upload |
| `MirrorOperations` | Start operations from saved configs, real-time log streaming, stop/delete controls |
| `History` | Filterable list of past operations with CSV export |
| `Settings` | Pull secret management, registry authentication verification, cache cleanup, catalog sync |

State is managed locally within each component via React hooks. Cross-cutting concerns use React Context:
- `AlertContext` — global alert notifications
- `ThemeContext` — light/dark/system theme toggle

### Backend (Express / Node.js)

The backend is an Express server running on Node.js 22 via `tsx` (TypeScript execution). It serves the REST API and, in production, the static frontend build.

Key responsibilities:

**Config Management** — CRUD operations for `ImageSetConfiguration` YAML files stored on disk. Validates YAML structure (kind, apiVersion, mirror section) on save/upload.

**Operator Catalog Data** — Serves pre-fetched operator metadata (operators, channels, versions, dependencies) from JSON files generated at build time by `sync-catalogs.sh`. Falls back to built-in catalog data if no runtime sync has been performed.

**Operation Execution** — Spawns `oc-mirror` as a child process with `--v2` flag for mirror-to-disk workflows. Tracks running processes, streams logs to files, and delivers them to the browser via SSE (`/api/operations/:id/logstream`). Handles process lifecycle (start, stop via SIGTERM/SIGKILL, completion detection).

**System Management** — Pull secret detection and CRUD, registry authentication verification (OAuth2 token exchange), cache size reporting and cleanup, catalog sync execution.

### Catalog Data Pipeline

Operator catalog metadata follows this pipeline:

```
registry.redhat.io catalogs
       |
  sync-catalogs.sh              <- Extracts FBC data, generates operator/channel/dependency JSON
       |
  catalog-data/                 <- Built-in (bundled at image build time)
  data/catalog-data/            <- Runtime (generated by in-app sync)
       |
  Express server                <- Loads into memory, serves via API
       |
  React frontend                <- Offline operator browsing, channel selection, dependency detection
```

The sync script processes three catalog types (Red Hat, Certified, Community) across supported OCP versions (4.16–4.22), producing:
- `catalog-index.json` — catalog registry with URLs and digests
- `operators.json` — per-catalog operator list with channels and versions
- `dependencies.json` — per-catalog operator dependency mappings

### Container Architecture

The Dockerfile uses a multi-stage build:

| Stage | Purpose |
|-------|---------|
| `builder` | Installs npm dependencies, copies catalog data, runs `vite build` |
| `downloader` | Fetches the oc-mirror v2 binary for the target architecture (AMD64/ARM64) |
| `production` | Minimal Node.js runtime with production dependencies, built frontend, oc-mirror binary, and catalog data |

The production container runs as a non-root user and exposes port 3001 internally. Both `mirror-gui.sh` (the primary run entrypoint) and `local-build.sh` (build-and-run for development) map it to port 3000 on the host, falling back to the next available port if 3000 is in use.

## Key Data Structures

**ImageSetConfiguration** — The oc-mirror input YAML that defines what to mirror: platform channels, operator catalogs with package/channel/version filters, additional images, and helm charts. Mirror-GUI builds this visually and validates it before saving.

**OperationRecord** — Server-side JSON representing a mirror operation: ID, config file, status (running/success/failed/stopped), timestamps, duration, logs, and error messages. Persisted as individual JSON files.

**PreFetchedCatalogData** — In-memory representation of all pre-fetched operator catalog metadata, keyed by catalog type and OCP version. Includes operator lists, channel mappings, and dependency data.

## REST API

The API is organized into endpoint groups:

| Group | Prefix | Purpose |
|-------|--------|---------|
| System | `/api/health`, `/api/system/*` | Health checks, system info, available paths |
| Config | `/api/config/*` | CRUD for ImageSetConfiguration YAML files |
| Catalogs | `/api/catalogs`, `/api/operators/*` | Operator catalog browsing, channels, versions, dependencies |
| Operations | `/api/operations/*` | Start/stop/delete operations, logs, log streaming (SSE) |
| Pull Secret | `/api/pull-secret/*` | Pull secret CRUD |
| Registry | `/api/registries/*` | Registry listing and auth verification |
| Cache | `/api/cache/*` | Cache cleanup |
| Catalog Sync | `/api/catalogs/sync*` | Trigger and monitor catalog data refresh |

Full API documentation is in [API.md](API.md).
